02. サブPCから何をどう同期するか

メインPCの tools/export_live_today.py が、SSHホスト別名 sub を使って サブPCのライブディレクトリを参照します。現在のLANではメインが 192.168.0.37、 サブが 192.168.0.238 です。どちらもインターネットから直接到達しないプライベートIPです。

成果物生成

サブPCのstack2tanが、スケジュール、推論、買い目、送信レシート、精算結果を出力。

対象検出

メインPCがSSHで当日稼働しているエンジン名とファイルを確認。

差分転送

rsync -azで必要なファイルをメインPCの一時キャッシュへpull。

公開用変換

Pythonが必要項目だけを抽出し、暫定精算や返還処理も加えて単一JSONへ整形。

画面更新

ブラウザが20秒ごとにJSONを取得し、再ビルドなしでライブ表示を更新。

なぜSSH＋rsyncなのか

SSH

通信を暗号化し、公開鍵認証とホスト別名で接続を管理できます。FTP用の別サービスを増やす必要がありません。

rsync

毎回全コピーせず差分だけ転送できます。ファイルベースの既存artifactと相性がよく、実装も運用も単純です。

重要な設計判断： 同期方向は「メインPCがサブPCからpull」です。 可視化側からサブPCへ書き込まないため、ポートフォリオ画面の不具合が推論・投票系へ波及しにくい構成です。

03. サーバーはどこに立ち、バックエンドは何か

WebサーバーはメインPCのWindows上ではなく、Windows内のWSL2 Linuxで動きます。 現在は公開パスを限定したPythonサーバーをsystemdユーザーサービスとして起動し、 ループバックアドレスのTCP 8000番で配信しています。 

systemctl --user status boatrace-public

# origin（外部から直接到達できない）
http://127.0.0.1:8000
Web配信層tools/serve_public.py。トップページとライブJSONだけを許可し、その他は404にする読み取り専用サーバー。
データ整形層Python exporter。SSH/rsync、artifact解析、返還を含む暫定精算、JSON生成を担当。
データ生成層サブPCのstack2tan。予測モデルと実運用ロジックは公開サイトから分離。
表示層ブラウザ内のJavaScript＋Canvas。JSONを読み、地図、推論、確率、収支を可視化。
「APIバックエンド」と言い切らない： 現状、Webリクエストに応じて計算するFlask/FastAPI等はありません。 Python exporterが事前にJSONを作る「バッチ＋静的配信」です。小規模な読み取り専用ダッシュボードとして、 障害点と依存を減らすための意図的な構成です。

05. Windows・WSL2・Firewallの関係

WSL2はWindows上で動く軽量Linux仮想環境です。このPCではLinuxカーネル microsoft-standard-WSL2とsystemdが有効です。アプリがLinux側で8000番をlistenしても、 LANから届くまでには複数の境界があります。

ブラウザサブPCまたは外部の閲覧者がURLへ接続。
LAN / Router家庭内ネットワーク。インターネット公開ではNATが受信を遮断。
Windows FirewallWindowsホストへの受信TCPを規則で許可・拒否。
WSL2 NetworkWindowsとLinuxの間を転送。NAT型・mirrored型等で到達経路が変わる。
Linux process127.0.0.1:8000だけで待ち受ける公開専用Webサーバー。

初期検証ではメインWSL2に 192.168.0.37 が見え、サーバーを 0.0.0.0:8000で待ち受けても、 サブPCからの直接アクセスはタイムアウトしました。つまりLinuxプロセスの待受だけでは不十分で、 Windows FirewallまたはWSL転送境界で受信が止まっていたと判断できます。現在は受信ポートを開けず、 サブPCにはSSH逆トンネル、外部にはCloudflare Tunnelを使っています。

大事な理解： localhost、LAN公開、インターネット公開は別物です。 自分のPCで見えることは、別PCから到達できることを意味しません。

06. サブPCから見せるSSH逆トンネル

Windows Firewallの受信設定を変更せずにサブPCから確認するため、メインPCからサブPCへSSH接続を開始し、 その接続の中に逆向きのHTTP経路を作りました。 

ssh -fNT \
  -o ExitOnForwardFailure=yes \
  -o ServerAliveInterval=30 \
  -o ServerAliveCountMax=3 \
  -R 8000:localhost:8000 sub
指定意味
-R 8000:localhost:8000サブPC側の8000番を、SSH越しにメインPC側のlocalhost:8000へ転送。
-Nリモートコマンドは実行せず、トンネルだけ作る。
-T疑似ターミナルを割り当てない。
-f認証後にバックグラウンドへ移動し、VS Codeのターミナル終了から切り離す。
ServerAlive*切断を検知しやすくする。恒久的な自動再接続そのものではない。
MAIN WSL2localhost:8000
  • Webサーバー
  • SSH接続を外向きに開始
⇄
ENCRYPTED SSHReverse Forward
  • HTTPをSSH内にカプセル化
  • Windows受信8000番を直接通らない
⇄
SUB PClocalhost:8000
  • ブラウザはローカルとして接続
  • 実体はメインPCから配信

VS Code再起動で落ちたのは、最初のSSHプロセスがターミナルに紐づいていたためです。 -fでバックグラウンド化すればVS Code終了には耐えます。PC再起動・回線断からも自動復旧させるなら、 systemd user serviceまたはautosshで常駐化します。

08. 公開時に実施した防御と次の改善

公開範囲の限定は実施済みです。 初期のpython -m http.serverでは.gitやtoolsまで取得できることを外部テストで発見しました。 現在はallowlist方式へ切り替え、/、/index.html、 /data/live_today.json以外を404にしています。
  1. 実施済み： originを127.0.0.1だけにbindし、外部から直接アクセスさせない。
  2. 実施済み： Named Tunnelとnemui3900.comを使い、固定URLとsystemd自動再起動を構成。
  3. 実施済み： dotfileとツールを非公開化し、MIME、キャッシュ、基本セキュリティヘッダーを明示。
  4. 秘密情報を分離： SSH鍵、生ログ、モデル、認証情報、送信payloadを公開ディレクトリへ置かない。
  5. JSONを最小化： ポートフォリオ表示に必要なデータだけをexportし、個人情報・内部パスを含めない。
  6. 今後： 同期ループも永続サービス化し、停止時は古い値をLIVE表示しない監視を追加する。

09. 面接で聞かれそうな質問

Q. なぜ2台構成ですか？

推論・投票系と表示系の負荷と障害範囲を分離するためです。表示側はサブPCから読み取るだけにし、公開サイトから実運用系への書き込み経路を持たせていません。

Q. なぜDBやAPIを使わないのですか？

更新頻度が分単位で、クライアントは読み取り専用だからです。単一JSONのスナップショットで要件を満たし、APIサーバーやDBの運用コストを避けました。

Q. SSHトンネルとは？

SSHの暗号化接続内で別のTCP通信を転送する仕組みです。今回は逆トンネルにより、サブPCのlocalhostをメインPCのWebサーバーへ接続しました。

Q. Cloudflare Tunnelの利点は？

自宅側から外向き接続を張るため、ルーターのポート開放なしでHTTPS公開できます。公開入口と証明書管理をCloudflare側へ寄せられます。

Q. なぜWebSocketではない？

元データの生成が60秒周期なので20秒ポーリングで十分です。リアルタイム性の必要以上に双方向常時接続を導入せず、再接続や状態管理を単純にしました。

Q. TypeScriptを使っていますか？

現状はVanilla JavaScriptです。初期のゼロビルドと試作速度を優先しました。データ型と画面状態が増えたため、次の保守段階ではTypeScript＋schema validationへの移行価値があります。

Q. セキュリティは？

サブPCへの同期は公開鍵SSHのpullのみ。外部公開は受信ポートを開けないNamed Tunnel方式です。originはloopback限定で、allowlistにないdotfileやツールは404になります。

Q. 一番の学びは？

アプリがlocalhostで動くことと、別端末・外部から安全に到達できることは別問題です。OS、仮想化、Firewall、NAT、トンネルを層として切り分けて検証しました。

60秒版の説明例

「このサイトは、サブPCで実運用している3連単AIの状態を見せるポートフォリオです。 推論成果物をメインPCのWSL2からSSHとrsyncで読み取り専用同期し、Pythonで公開用JSONへ整形しています。 フロントはVanilla JavaScriptとCanvasで、20秒ごとにJSONを取得して可視化します。 LAN確認ではSSH逆トンネル、外部公開ではCloudflare Tunnelを使いました。 そのためWindows Firewallや自宅ルーターに受信ポートを開けずにHTTPS公開できます。 originはloopbackだけで待ち受け、公開パスもトップページとライブJSONへ限定しています。」
